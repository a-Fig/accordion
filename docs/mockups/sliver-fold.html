Same block data rendered both ways. Left: today's treatment (folded = dimmed, hatched, full-size square). Right: the proposal — a folded block squeezes to a thin vertical sliver inside its cell slot (column grid never reflows); its digest/summary appears as a real summary block right before them; the whole run packs into ONE compact cluster — summary cell + tightly butted slivers — held by a soft calm bubble (no dead space; it wraps and holds, it never points). Slivers carry token weight as horizontal lines (count = die face); summary blocks carry a die face for their own size.

The tradeoff now: packing the slivers kills the dead space, so a folded run is smaller than the live squares it replaces — a 4-block fold ≈ one summary cell + a thin 4-bar strip, vs four full squares. What you give up is strict per-block grid alignment: a folded run no longer sits on integer cell columns, it becomes its own compact unit in the flow. That's the cost of "no dead space." The existing collapse-to-one-brown-tile group still exists for when you want the run gone entirely, not just compacted.
